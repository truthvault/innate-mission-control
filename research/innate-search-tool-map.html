Scenario routing map

How Hermes should choose tools in real Innate work

The right answer is not “use every tool every time.” It is “use the cheapest trustworthy lane that matches the job, then escalate only when confidence or coverage demands it.”

Routing principle

For Innate, source-of-truth accuracy matters more than search cleverness. Customer/order/quote/payment/workshop facts start internal. Public-market, SEO and competitor questions start external. Exact page evidence comes from extraction. Visual claims come from browser inspection.

Default: local first for opsExa first for discoveryBrave/Tavily for public sanityExtract before claiming

Stop rules

  • If the question is customer/order/payment-specific, do not use web as authority.
  • If the result will be customer-visible or live-site-visible, draft/inspect only until Guido approves.
  • If search results disagree, extract the original pages or label uncertainty.
  • If a browser screenshot or page state is broken, say so rather than pretending visual proof exists.

Recommended Hermes setup

Practical configuration and routing policy

1. Keep named lanes

Do not hide all search behind one generic command. Hermes should expose or internally route to named lanes: local truth, Exa semantic, Brave independent search, Tavily broad search, web_extract, Firecrawl crawl, browser visual.

2. Treat current web_search as Exa-backed

Config shows web.search_backend: exa. That is excellent for discovery, but it means ordinary web_search is not an independent Brave/Tavily sanity check unless routed explicitly.

3. Add a Firecrawl decision gate

Direct Firecrawl was unavailable because no configured credential was present. Keep it optional until site-wide crawls, supplier catalogue extraction or structured scrape jobs justify it.

4. Evidence before strategy

For reports, use search to choose sources, then web_extract to capture claims. Search snippets are leads, not proof.

5. Browser only when it earns its cost

Use browser for live website layout, JavaScript, navigation, form and visual QA. Do not spend browser cycles on simple facts already available via extraction.

6. Internal truth beats web

For live customer, order, quote, invoice, payment or production claims, use the approved internal sources and read back proof. Web content can support product language only.

Recommended default policy

1
Is this about a customer/order/quote/payment?

Use internal systems first. Stop if exact evidence is missing.

2
Is this broad research or SEO?

Use Exa first for semantic discovery, then Brave/Tavily to check public visibility.

3
Do we need to quote or rely on a page?

Use web_extract on the original URLs. If the site needs crawling and Firecrawl is configured, use Firecrawl.

4
Does layout, interaction or visual brand matter?

Use browser automation/vision. If screenshots fail, report that limitation and use DOM evidence only.
